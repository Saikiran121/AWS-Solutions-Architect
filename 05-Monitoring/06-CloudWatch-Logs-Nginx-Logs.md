# Hands-On Lab: Streaming Nginx Application Logs to CloudWatch

This lab continues from the base CloudWatch Agent installation lab. In the previous lab, we streamed a single system file (`/var/log/syslog`). 

However, in the real world, an EC2 instance might run an application (like Nginx) that generates *multiple* log files. This lab demonstrates how to modify the CloudWatch Agent's JSON configuration array to simultaneously stream different application logs to entirely different CloudWatch Log Groups.

---

### Step 1: Install Nginx on the EC2 Instance

First, we need the application that will generate our custom logs. Connect to your Ubuntu EC2 instance via SSH.

1.  **Install the Nginx web server:**
    ```bash
    sudo apt update -y
    sudo apt install nginx -y
    ```

---

### Step 2: Confirm Nginx Log Locations

Before telling the CloudWatch Agent to fetch a log file, you must verify exactly where the application stores those files on the hard drive. 

1.  List the contents of the default Nginx logging directory:
    ```bash
    ls -l /var/log/nginx/
    ```
2.  *You should see two critical files:*
    *   `/var/log/nginx/access.log` (Tracks every user who visits your website).
    *   `/var/log/nginx/error.log` (Tracks 404s, 500s, and server configuration issues).

---

### Step 3: Edit the CloudWatch Agent Configuration

We need to update our existing JSON file. We will expand the `collect_list` JSON array to add two new specific configuration blocks for Nginx.

1.  Open your existing CloudWatch Agent configuration file using `nano`:
    ```bash
    sudo nano /opt/aws/amazon-cloudwatch-agent/bin/config.json
    ```

2.  Modify the file to look exactly like the JSON below. Notice how we are appending the specific `"timezone": "UTC"` parameter for Nginx logging accuracy:
    ```json
    {
      "logs": {
        "logs_collected": {
          "files": {
            "collect_list": [
              {
                "file_path": "/var/log/syslog",
                "log_group_name": "/ec2/syslog",
                "log_stream_name": "{instance_id}",
                "timezone": "UTC"
              },
              {
                "file_path": "/var/log/nginx/access.log",
                "log_group_name": "/ec2/nginx/access",
                "log_stream_name": "{instance_id}",
                "timezone": "UTC"
              },
              {
                "file_path": "/var/log/nginx/error.log",
                "log_group_name": "/ec2/nginx/error",
                "log_stream_name": "{instance_id}",
                "timezone": "UTC"
              }
            ]
          }
        }
      }
    }
    ```
    *(**Explanation:** We are now commanding the agent to track three separate files and send them to three distinctly named Log Groups in the AWS Console).*

3.  Save and exit `nano` (Press `CTRL+O`, `Enter`, then `CTRL+X`).

---

### Step 4: Restart the CloudWatch Agent

Whenever you modify the `config.json` file on disk, you must force the agent to read the new configuration.

1.  Execute the fetch-config command again to apply the new file:
    ```bash
    sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
    -a fetch-config \
    -m ec2 \
    -c file:/opt/aws/amazon-cloudwatch-agent/bin/config.json \
    -s
    ```
    *The agent will restart invisibly in the background and immediately look for the Nginx files.*

---

### Step 5: Generate Nginx Traffic

Log streaming doesn't work if the files are empty. Let's create some fake traffic to force Nginx to write entries to the `access.log`.

1.  **Hit the local server from the command line:**
    ```bash
    curl localhost
    ```
2.  **Generate external traffic (Optional):**
    Open a web browser on your personal computer and navigate to `http://<Your-EC2-Public-IP>`. Refresh the page a few times.

---

### Step 6: Verify in the AWS Console

Let's verify that the CloudWatch agent successfully separated the different application logs into their designated folders.

1.  Navigate to **CloudWatch** in the AWS Console.
2.  On the left-hand menu, click **Logs** -> **Log groups**.
3.  You should now see two brand new Log Groups appear:
    *   `/ec2/nginx/access`
    *   `/ec2/nginx/error`
4.  Click on `/ec2/nginx/access`.
5.  Click on the Log Stream named after your Instance ID.
6.  You will now see the exact `GET` requests (generated by `curl` or your web browser) streaming directly into the AWS Console.
